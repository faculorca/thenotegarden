## 7. Specific Examples of Issues Found

### 7.1 Example: Redundant `type` and `categories` Properties

Many files use both `type` and `categories` to essentially categorize the same information.

**Example from Slip-Box note:**
```yaml
categories:
  - "[[Zettelkasten]]"
type:
  - "[[Main Note]]"
```

**Example from Book note:**
```yaml
categories:
  - "[[Zettelkasten]]"
  - "[[Books]]"
type:
  - "[[Reference Note]]"
  - "[[books]]"
```

**Issue:** Notice in the second example, "Books" appears in categories with brackets `[[Books]]` but in type it appears as `[[books]]` (lowercase) - this is a case inconsistency that makes querying difficult.

### 7.2 Example: Inconsistent Date Properties

There are multiple date-related properties being used inconsistently:

**`date` property variations:**
- As datetime object: `[datetime.date(2025, 11, 18)]`
- As string with time: `2025-09-26, 13:46`
- As string without time: `2025-11-05`
- As nested list: `[[datetime.date(2025, 11, 24)]]`
- As triple-nested list: `[[[datetime.date(2025, 11, 24)]]]`

**`created` property variations:**
- As date: `2025-11-05`
- As list with date: `[datetime.date(2025, 11, 6)]`
- As empty string: `` (64.3% of files with this property have it empty!)

**Both `date` AND `created` are used** in 277 and 196 files respectively, but it's unclear which one represents creation date vs publication date vs modified date.

### 7.3 Example: Empty Properties

Many properties are defined but left empty in a large percentage of files:

**High empty rates:**
- `via`: 100% empty (11/11 files)
- `end`: 100% empty (4/4 files)

**Recommendation:** These properties add noise to frontmatter. Either populate them or remove them from files where they're not needed.

### 7.4 Example: Property Value Format Inconsistencies

**`categories` property shows multiple formats:**
- String format: `[[People]]` (32 occurrences)

### 7.6 Example: Content Type Patterns

The `content type` property (206 files) shows a well-structured taxonomy:
```yaml
content type:
  - "[[Tip]]"
  - "[[LongForm]]"
  - "[[Educate Me]]"
```

This is actually a GOOD pattern - it uses three dimensions:
1. Content format (Tip, Observations, Engagement, etc.)
2. Length (LongForm, LinkedIn, Tweet, etc.)
3. Purpose (Educate Me, Challenge Me, Entertain Me)

**However**, this overlaps with both `type` and `categories`, creating confusion.

- `['categories']`
- `['to-read']`

**This indicates** that the vault primarily uses frontmatter properties rather than inline tags for organization, which is good for consistency.

**Content organized by type:**

- Files in `Categories/People/` also have `categories: [[People]]` property

- Templates have template-specific properties but are also in a Templates folder
